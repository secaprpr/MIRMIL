# RRT-MIL survival reference plan

Date: 2026-07-16

Purpose: run the external RRT-MIL survival implementation and its survival baselines on the already prepared MIRMIL prognosis feature datasets, without modifying MIRMIL's environment, extracted features, or prognosis model code. This is a diagnostic reference to test whether MIRMIL prognosis underperformance is caused by our prognosis code/protocol or by the data/features/task.

## RRT-MIL survival code facts

- Entry point: `/data15/data15_5/fanhao/projects/reference/RRT-MIL/Survival/main.py`
- Dataset class: `Survival/datasets/TCGA_Survival.py`
- Expected CSV columns: `Study, ID, Event, Status, WSI`
- `Status=1` means event observed; `Status=0` means censored.
- Internally, RRT-MIL converts `Status=0` to `Censorship=1`.
- Survival time is read from `Event`.
- Patient WSI features are loaded from semicolon-separated `.pt` paths in `WSI`.
- The official implementation performs its own fixed 5-fold random CV using `random.seed(1)`.
- The official implementation does not use our existing train/val/test split.
- Validation metric is C-index from `sksurv.metrics.concordance_index_censored`.
- Risk score is `-sum(S)`, where `S` is the predicted survival curve.
- Default loss is `nll_surv`.

## Important comparability issue

Directly running RRT-MIL's official 5-fold CV is not directly comparable to MIRMIL prognosis results if MIRMIL used our existing train/val/test split.

Therefore results must be separated into two protocols:

1. `rrtmil_official_5fold`
   - Uses RRT-MIL's original fixed random 5-fold CV.
   - Useful for checking whether the reference code runs and whether baseline performance is plausible.
   - Not a fair direct comparison to MIRMIL split-based results.

2. `mirmil_matched_split`
   - Minimal adapter should force RRT-MIL baselines to use our existing train/val/test split.
   - This is required before concluding that MIRMIL prognosis code is correct or buggy.

## Execution plan

### Phase 1: official RRT-MIL baseline sanity check

Run official 5-fold CV for a small baseline set first:

- KIRC OS, UNI: `MeanMIL`, `AttMIL`, `TransMIL`, `RRTMIL`
- KIRC OS, R50: `MeanMIL`, `AttMIL`, `TransMIL`, `RRTMIL`
- BLCA OS, UNI/R50 after KIRC sanity check

Use this only as a reference sanity check, not as the final fair comparison.

### Phase 2: matched-split adapter

Implement a minimal RRT-MIL survival adapter that:

- reads the `Split` column already present in converted CSVs;
- preserves RRT-MIL models, loss, risk computation, and optimizer;
- reports train/val/test C-index on the same split used by MIRMIL;
- saves reproducible commands and result TSV rows.

This phase is required before diagnosing MIRMIL prognosis bugs.

### Phase 3: matched-split diagnostic matrix

Run the following models under the matched split:

- `MeanMIL`
- `MaxMIL`
- `AttMIL`
- `CLAM_SB`
- `DSMIL`
- `TransMIL`
- `RRTMIL`

Datasets/features:

- KIRC OS: R50, UNI
- KIRC PFS: R50, UNI
- BLCA OS: R50, UNI

Optional after initial matrix:

- `DTFD`
- `MHIM-MIL`
- `CLAM_MB`
- `IBMIL` only if its confounder/clustering prerequisites are satisfied.

### Phase 4: interpretation

Decision rules:

- If RRT-MIL matched-split baselines are clearly stronger than MIRMIL on the same data/features/split, inspect MIRMIL prognosis loss, censoring, risk sign, time units, C-index, and split handling first.
- If RRT-MIL official 5-fold is high but matched split is not, split difficulty or protocol mismatch is the likely cause.
- If all matched-split baselines are weak, the issue is more likely feature/task/data-related than a MIRMIL-only code bug.
- If UNI strongly beats R50, prognosis signal may depend on high-level morphology.
- If both R50 and UNI are unstable, inspect event count, censoring ratio, patient count, and split robustness.
